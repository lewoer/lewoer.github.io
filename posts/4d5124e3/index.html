Mastering Java Exceptions

发表于 2019-01-14 | 分类于 翻译 | | 热度 ℃

[翻译]Mastering Java exceptions, Part 1

英文原文

java exception是库类型，而且也是一种用来代表和处理程序异常的语言特性。在这篇文章的第一部分你会学到自java1.0以来的一些基本的语言特性和库类型。第二部分你会学到一些当前的高级特性。如果你想理解程序异常如何在源码中运行，那么你就来对地方了。除了介绍java exception，我也会介绍throwing objects, try code, catch thrown objects 而且在你抛出异常之后清理自己的代码。

什么是java exception？

当一个程序的正常运行被一些意想不到的行为打断的时候，一些故障就会发生。而这种分歧(divergence)称之为exception。举例来说，一个程序尝试打开以恶搞文件来读取内容，但是这个文件却不存在。java 将exception分类为好几种，所以我们一个个来认识。

checked exceptions

java分类exception产生于一些外部因素(比如说缺失文件)为checkedxceptions.java编译器检查这些exception要么当它们发生的时候就处理了，或是在其他地方标记为已经处理。

Exception handlers
execption hander是一段处理异常的代码。当异常发生的时候，这段异常处理程序读取了存储在作用域内的变量的数值。然后用它了解到的信息去恢复java程序为正常状态。比如说，一个exception handler可能读取了存储的文件名，然后提醒用户去替换缺失的文件。

Runtime (unchecked) exceptions

You might modify a program to handle a runtime exception, but it’s better to fix the source code. Runtime exceptions often arise from passing invalid arguments to a library’s methods; the buggy calling code should be fixed.
假定一段程序想要以整数除以整数，这种不可能的事情正好阐述了另一种类型的异常，runtime exception.不像checked exception，runtime exception一般来源于差劲的代码，而且应该被程序员修正。因为编译器不会检查runtime exception已经被处理了或是在其他地方被记录为处理。可以认为runtime exception为unchecked exception.

关于runtime exception
你可能会修改程序来解决runtime exception，但是最好去修改源代码。runtime exception经常来源于传递非法参数到以一个库方法，错误的调用代码应该被修复。

Errors

一些exceptions非常严重，一位它们破环了程序继续执行的能力。比如一个程序想要从JVM分配内存，但是这里没有足够的内存用来满足需求。另外一种严重的情况是，当一个程序企图通过一个Class.forName()方法调用来装载类文件，但是类文件是损坏的。这种类型的异常通常为error，你不应该尝试过自己处理error，因为JVM可能不会恢复它。

Error codes versus objects

C语言使用基于整数的error code来代表故障。

C语言的chdir函数成功返回0，失败返回-1.同理，fopen成功时候向FILE 结构返回非空指针，否则返回一个null。不论发生什么情况，你必须读取全局errno变量（基于整数的error code）

错误代码存在以下几个问题：

  • 整数是无意义的，它们不知道它们代表什么异常，比如说6.
  • 将上下文于错误代码关联起来很尴尬。你想要输出不能打开的文件名字，但是你去哪里存储文件名字。
  • 整数是任意的，但阅读源码的时候会招致困惑。比如，指定if (!chdir("C:\\temp"))而不是if (chdir("C:\\temp"))来测试异常是很清楚的，但是0用来选择表明成功，所以if (chdir("C:\\temp"))必须表示失败。
  • 错误代码很容易忽视，容易造成bug。程序员指定了chdir("C:\\temp");但是却忘记了非空检查if (fp == NULL),而且程序员不需要检查errno.

为了解决这些问题，java采用了一种新的异常处理方法。在java中，我们将描述exception的对象和基于throw,catch对象的机制结合再一起。优点是

  • 对象能够被以一个有意义的名字的类创建。FileNotFoundException(java.io package)就比6有意义。
  • 对象能过存储上下文在多个字段里。你可以存储一段消息，文件名字（无需打开文件），解析操作失败的最近位置，或其他一些item到对象字段。
  • 不用使用if来测试故障，exception对象抛出到异常处理器里面来与程序代码进行分离。代码更易读，不容易出现bug。

Throwable and its subclasses

java提供了类的层级结构来代表不同的exception。这些类都源于java.lang.Throwable类。

Throwable是关于exceptin的最终超类。只有从Throwable和它的子类创建的对象能够被抛出（和随后被catch），这些对象被称为throwables。

一个可以抛出的对象与描述exception的详细信息有关联。一些构造器包括下面描述的一对，被用来创建存储详细信息的Throwable对象。

  • Throwable()用来创建没有详细信息的Throwable对象。这个构造器适合于没有上下文的情况。比如说，你只想知道这个栈是空的还是满的。
  • Throwable(string message) 创建了有详细信息的Throwable对象。这个消息能够向用户输出或是能够记录在日志里。

Throwable提供了String getMessage()方法来返回详细信息。它也提供一些其他有用的方法，下面将会介绍。

The Exception Class

Throwable有两个直接的子类，其中一个是Exception，用来描述引发于外部因素的异常，比如尝试从不存在的文件进行读取。Exception声明了一个和Throwable相同参数列表的构造器，每一个构造器调用了它可以抛出的对应函数。Exception继承了Throwable的方法，而且美欧声明新的方法。

java提供了很多exception子类。

  • CloneNotSupportException表明尝试克隆一个没有实现cloneable接口的类的对象。两种类型都在java.lang
  • IOException表明发生了一些I/O故障。它在java.io里面。
  • ParseException表明在解析文本的时候发生错误。可以在java.text包找到。

每一个Exception子类后面都有Exception。这样一个惯例很容易区分类的目的。

Exception或他除了RuntimeException的子类实例化的对象都是checked exception。

RuntimeException

RuntimeException直接继承了Exception，它的异常来源于糟糕的代码。RuntimeException声明了一个和Exception相同参数列表的构造器，每一个构造器都调用了Exception的对用函数。RuntimeException继承了Throwable方法，没有声明新的方法。下面是它的子类。

  • ArithmeticException 表明错误的数学操作，比如整数除以0.
  • IllegalArgumentException表明非法或是不合适的参数传递到方法。
  • NullPointerException表明企图通过null引用来调用方法或是访问实例字段。

从RuntimeException或它的子类实例化的对象是unchecked exception

The Error class

Throwable的另外一个直接的子类error，它描述了一个反常的问题，正常的应用不应该尝试解决，比如说内存耗尽，溢出JCM堆栈，尝试装载不存在的类，像exception一样，Error声明了和Throwable一样的构造器，继承了throwable方法，没有声明新的方法。

你可以从它们类名以Error结尾的惯例中辨别Error子类。包括OutOfMemoryError, LinkageError,StackOverflowError,它们都属于java.lang

Throwing exceptions

C语言通知exception的调用代码是通过设置errno全局变量来返回错误代码。对比来说，java抛出一个对象。知道如何和在什么时候抛出异常是java编程的必要之处，抛出exception的两步。

  • 使用throw语句来抛出异常对象
  • 使用throw子句来通知编译器。

Checked exception争议
throw子句和checked exception是有争议的，许多开发者厌恶强制声明throws或是处理checked exception(s).Are checked exceptions good or bad?

通知编译器一个checked ClassNotFoundException被抛出很必要，它将throws ClassNotFoundException子句依附在调用的main()方法头上。毕竟，异常在这个方法里面并没有被处理。当异常抛向JVM时候，jvm将会记录。在这个例子中，因为main()没有父类方法，所以它会输出一段信息结束。

你之后会看到throws其他的例子。在处理throws子句时候，注意几条规则。

  • 如果可能的话，不要在一个throws子句里包含unchecked exception类名（如ArithmeticException）,因为throws子句是为了checked exception设计的。包含unchecked exception只会使源码更加混乱。
  • 你可以添加一个throws子句到构造器。当构造器执行的时候出现错误，这时从构造器抛出checked exception.生成的对象不会被创建。

  • 如果子类方法声明一个throws子句，重写的子类方法不必去声明throws子句。如果子类方法声明了一个throws子句，则子句一定不包括checked exception类的名字。同时超类方法的throws子句也不包括，除非它们是exception的子类名字。

  • 当checked exception的超类出现时候，它的名字就不必出现在throws 子句中了。因为已经包括在内了。

  • 当方法抛出checked exception时候，编译器报告错误。这时候无需解决异常或列出在throw子句中的异常。
  • 你应该在一个方法的throws子句中声明checked exception类名，而无需从这个方法抛出这个类的实例。

确保checked exception可读性
有时候，一个方法抛出很多的checked exception，而且很容易指定以恶搞throws exception子句来保存keystrokes，虽然仅仅指定Exception很省时间，但是它是的源码不具备可读性，建议只关注于私有的，一次性代码。

JDK 1.0和JDK 1.4中异常处理：堆栈跟踪 stack traces

每一个JVM线程（一个执行路径）与栈有联系，当线程创建的时候，栈随即也被创建。这个数据结构被分为很多桢，它的数据结构与方法调用有联系，因为这个原因，每一线程的堆栈通常称为方法调用堆栈。

每次调用方法的时候都会创建一个桢，每个桢存储本地变量，参数变量（保存传递到方法的参数），返回到方法调用的信息，存储在返回值的空间，对调度异常的信息等等。

堆栈追踪或是栈回溯是一个活动的堆栈桢的报告，它在线程执行期间内的一个特定时间，一个特定的点。java的Throwable类提供了一些方法打印stack trace,填充 stack trace和访问stack trace 元素。
